# Vendoring notes

Vendored from
[`Azure/terraform-azurerm-avm-ptn-cicd-agents-and-runners`](https://github.com/Azure/terraform-azurerm-avm-ptn-cicd-agents-and-runners)
at tag **`v0.5.2`** into `terraform/modules/cicd-runners/` to keep
everything under `terraform/modules/` local, allow repo-wide version-pin
alignment, and permit the Key Vault-reference patch for the GitHub App
private key (which upstream does not yet support).

`README.md` is regenerated by `terraform-docs -c .terraform-docs.yml .`
from the source files and reflects the full surface, including local
extensions.

Upstream layout carries three submodules (`container-app-job`,
`container-instance`, `container-registry`); all three are vendored
verbatim with the same extension rules applied at the appropriate level.

## Local extensions vs upstream

### 1. Telemetry stripped

- Removed `main.telemetry.tf` (both root and submodules, where present).
- Removed `variable "enable_telemetry"` from root `variables.tf` and
  `modules/container-registry/variables.tf`.
- Removed the `modtm` provider from `terraform.tf` (root and submodules).
- Removed all `enable_telemetry` pass-throughs (module call args and
  `main.container.registry.tf` wiring).

### 2. Provider pins

Rewritten to the repo-wide convention (pessimistic-minor, per AGENTS.md
§6):

- Terraform `~> 1.14`
- `hashicorp/azurerm ~> 4.20`
- `Azure/azapi ~> 2.9`
- `hashicorp/random ~> 3.8`
- `hashicorp/time ~> 0.13`

### 3. `azurerm` carveout

This is the **only** module in the fleet repo allowed to use the
`azurerm` provider. Every callsite (e.g. `bootstrap/fleet/main.runner.tf`)
still uses `azapi` exclusively; `azurerm` is confined inside this
vendored tree because the upstream AVM module relies on it heavily
(`azurerm_container_app_job`, `azurerm_private_dns_*`, `azurerm_nat_*`,
the `avm-res-containerregistry-registry` child module, etc.). Mirrors
the existing `azuread` carveout for `modules/github-repo/`.

### 4. Key Vault reference for the GitHub App private key

Two new optional inputs on the **root** module and the
**container-app-job** submodule:

- `github_app_key_kv_secret_id` — versionless Key Vault secret URI
  (`https://<vault>.vault.azure.net/secrets/<name>`) holding the
  GitHub App private key (PEM). Container Apps resolves the latest
  version at runtime.
- `github_app_key_identity_id` — resource ID of the UAMI that Container
  Apps uses to read the secret (must have `Key Vault Secrets User` on
  the target vault and be attached to the job).

When `github_app_key_kv_secret_id` is set:

- Upstream validation on
  `version_control_system_github_application_key` is relaxed so the
  plaintext PEM input can stay `null`.
- In `locals.container.app.job.tf` (root), the `github_app` entry of
  `sensitive_environment_variables` keeps
  `container_app_secret_name = "application-key"` but sets
  `value = null`.
- In `modules/container-app-job/locals.tf`, the
  `secret_environment_variables` filter is relaxed to keep secrets
  whose `container_app_secret_name` is KV-backed even though
  `value == null`, and the emitted `secrets` list is split so KV-backed
  names render as `{ name, keyVaultUrl, identity }` (Container Apps
  Key Vault reference) instead of `{ name, value }`.

Result: the PEM lives only in Key Vault, is resolved by Container Apps
at runtime via the attached UAMI, and never materialises in Terraform
state. This is the minimum change needed because
`azurerm_container_app_job.secret.value` does not (as of
`azurerm 4.69.0`) support `write_only` / `value_wo`.

### 5. Dropped files

Not copied from upstream: `tests/`, `examples/`, `avm`, `avm.bat`,
`avm.ps1`, `Makefile`, `AGENTS.md`, `CODE_OF_CONDUCT.md`,
`CONTRIBUTING.md`, `SECURITY.md`, `SUPPORT.md`, `LICENSE`. The upstream
`LICENSE` (MIT) still applies — see the footer of the generated
`README.md`.

## Callers inside this repo

- `terraform/bootstrap/fleet/main.runner.tf` — the single shared
  repo-scoped GitHub runner pool used by fleet workflows. Uses GitHub
  App authentication with the private key sourced from Key Vault via
  the inputs documented in §4 above.
